- [SEV2] improvement/brief.py:93 — `web_search` tool registered without `max_uses` → unbounded search call fan-out per brief request. At hundreds of concurrent creators this is a real cost amplifier (each search round-trip is billed). Rubric §5 (Anthropic SDK). **Status: OPEN since Wave-4; re-verified 2026-06-01.** | fix: pass `"max_uses": settings.ANTHROPIC_WEB_SEARCH_MAX_USES` in the tool dict at line 93; add `ANTHROPIC_WEB_SEARCH_MAX_USES: int = 5` to `config.py` + `.env.example`. - [SEV2] improvement/brief.py:161-167 (and streaming path 139-147) — `tool_choice` not set on either the `.create()` call or the streaming `stream_and_emit()` call, so it defaults to `auto` and the model can skip `web_search` entirely. Rubric §5. **Status: OPEN since Wave-4; re-verified 2026-06-01.** | fix: pass `tool_choice={"type": "tool", "name": "web_search"}` (force at least one search) to both call paths; add regression test asserting response contains at least one `tool_use` block. Plumb `tool_choice` through `worker/anthropic_stream.py` as a new kwarg.

- [SEV2] improvement/brief.py:7-8, 58-60, 80-89 — docstring states "the static prefix is below Sonnet 4.6's minimum cacheable size, so the cache does not engage for this low-frequency call." The `cache_control: {"type": "ephemeral"}` marker at line 88 is therefore a 1.25× write premium for cache writes that will never be read. Rubric §5. **Status: OPEN since Wave-4; re-verified 2026-06-01.** | fix: either pad `_SYSTEM_INSTRUCTIONS` past ~1024 tokens (mirror `dna/brief.py`'s longer preamble), OR drop the `cache_control` marker entirely until the prefix grows. Close related decision entry in `docs/DECISIONS.md`.

- [BLOCKER] routers/improvement.py:109 and models.py:669-700 — comment at line 109 claims "`UNIQUE(creator_id)` backstop guarantees at most one of two truly-concurrent inserts succeeds," but the `ImprovementBrief` model definition (models.py:669-700) does NOT contain this constraint, and migration 0009_improvement_briefs.py does not create it. Without this constraint, the claimed debounce safety (lines 76-84) has a gap: if two concurrent POST requests both SKIP LOCKED and both find no row (because neither acquired a lock yet), they will both reach line 113 and `session.add()` different row instances. If both succeed at insert, two rows exist per creator. The `SELECT FOR UPDATE SKIP LOCKED` mechanism (lines 85-91) only prevents double-fire IF a row already exists; on the first POST after a completed brief, the second caller's SKIP LOCKED returns nothing, and without a UNIQUE constraint enforcing singleton-per-creator, both can insert. Rubric §2 (concurrency & scale). **Status: BLOCKER — ships double-fire risk at scale.** | fix: (1) add `__table_args__ = (sa.UniqueConstraint("creator_id", name="uq_improvement_briefs_creator_id"),)` to models.py after line 700; (2) file a new migration (0016_improvement_brief_unique) that adds `ALTER TABLE improvement_briefs ADD CONSTRAINT uq_improvement_briefs_creator_id UNIQUE (creator_id)`; (3) wrap the insert logic (lines 113-114) in a try/except that catches `IntegrityError` and re-queries for the existing row created by the concurrent request. Regression test: spawn two concurrent POSTs from different threads; both should return 202 with the same task_id.

- [cleanup] improvement/brief.py:134 — `from worker.anthropic_stream import stream_and_emit` is an in-function import without a "why" comment. The local import IS load-bearing (avoids circular import with `worker.tasks`). Rubric §6. **Status: OPEN since Wave-4; re-verified 2026-06-01.** | fix: add a one-line comment above the import: `# Local import: avoids circular dependency with worker.tasks`.

## Module verdict

**NEEDS-WORK — has BLOCKER.** The missing `UNIQUE(creator_id)` constraint on `improvement_briefs` table creates a double-fire hazard on concurrent POSTs. The `SELECT FOR UPDATE SKIP LOCKED` mechanism is correct as a guard for updates to existing rows, but it cannot prevent the first-POSTever race where two concurrent requests both SKIP LOCKED, both find nothing, and both succeed at inserting separate rows per creator. This is a real ship-blocking defect. The 3 SEV2s on Anthropic SDK (web_search without max_uses, no tool_choice, inert cache write) are cost/grounding defects but non-critical for launch. The 1 cleanup is very-low-priority.
