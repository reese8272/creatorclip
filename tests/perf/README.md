# tests/perf/ — load testing + BLOCKER verification

Unit tests prove correctness; they cannot prove behavior under hundreds of
concurrent creators, and CI hits Postgres **directly** so it can never catch the
one failure mode that takes prod down: psycopg3 prepared statements breaking
behind **PgBouncer transaction pooling** (Issue 58 — fixed with
`prepare_threshold=None` in `db.py`, but unprovable without a real pooler).

This directory turns both into one command.

## Quick start — verify the BLOCKER fix (Issue 58)

```bash
pip install -r requirements-dev.txt   # provides locust on PATH
./tests/perf/run.sh                   # 200 users / 20 spawn / 3m by default
```

`run.sh` brings up the production topology locally — **Postgres + PgBouncer
(transaction mode) + Redis + the app with its DB traffic routed through
PgBouncer:6432** — migrates, seeds a realistic creator, runs Locust against it,
then **scans the app logs for the prepared-statement failure signature**. If that
signature appears, the fix has regressed and `run.sh` exits non-zero. If it
doesn't, Issue 58 holds under transaction pooling.

Knobs:
```bash
USERS=400 SPAWN_RATE=40 RUN_TIME=5m ./tests/perf/run.sh
NO_TEARDOWN=1 ./tests/perf/run.sh     # leave the stack up to poke at
LOCUST=.venv/bin/locust ./tests/perf/run.sh
```

Outputs `tests/perf/perf_results_stats.csv` (+ `_stats_history.csv`) with the
percentile tables.

> **Tool choice:** Locust (Python) — reuses the project's auth scheme, same
> language as the codebase. If you later need >10k RPS from one box or native
> Prometheus streaming, k6 is the documented alternative (`docs/DECISIONS.md`).

## Files
| File | Role |
|---|---|
| `docker-compose.perf.yml` | The prod-shaped stack; PgBouncer is `POOL_MODE=transaction`. |
| `seed.py` | Idempotent seed of a fixed creator (UUID `…0000ff`) with videos/clips/DNA/embeddings/activity/balance so read paths return non-empty payloads. |
| `locustfile.py` | Authenticated dashboard session — read-heavy + a light write. Mints a session JWT in lock-step with `auth.py`. |
| `run.sh` | One-command orchestrator + the prepared-statement pass/fail check. |
| `.env.perf` | Generated by `run.sh` (gitignored) — fake external creds + a Fernet key. |

## Running against staging instead of the local stack

The local stack is for the BLOCKER check. To profile real staging latency, point
the locustfile at it directly (it needs the staging `JWT_SECRET_KEY` and a seeded
creator UUID):

```bash
export CC_BASE_URL=https://staging.example.com
export CC_JWT_SECRET=<staging JWT_SECRET_KEY>
export CC_CREATOR_ID=<seeded creator uuid>
locust -f tests/perf/locustfile.py --host "$CC_BASE_URL" \
       --users 300 --spawn-rate 20 --run-time 5m --headless \
       --csv docs/assessment/loadtest
```

## What to look for (feeds scale-checklist.md)
| Symptom in results | Likely cause | Checklist axis |
|---|---|---|
| `prepared statement … does not exist` in app logs | psycopg3 prepare under txn pooling — the BLOCKER (`run.sh` fails on this) | A |
| p99 climbs with users while p50 stays flat | event-loop stall — a sync/blocking call on a hot async path | B |
| Errors spike to `QueuePool limit` / `TimeoutError` | DB pool exhausted vs replicas × pool size | A |
| 429s appear earlier than expected | rate limiter; confirm per-creator not per-IP, and Redis round-trip cost | F |
| `/health` flips to `degraded` under load | DB or Redis saturation | A, E |
| Throughput plateaus then drops | a downstream (R2/YouTube/Anthropic) without a timeout backing up | E |
